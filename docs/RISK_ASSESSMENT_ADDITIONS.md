## Critical context-gathering: TerminalPanel vs. CapturePage tab patterns

**The user's stated understanding is partially incorrect and must be corrected before assessing Addition 4.**

### Fact: current state of `TerminalPanel`

`TerminalPanel` (`frontend/src/components/Terminal/TerminalPanel.tsx`) is mounted inside `TopologyPage` as a collapsible bottom drawer, controlled by `open`/`onToggle`/`height` props from the parent, with drag-resize handled by the parent via `bodyRef` (`TerminalPanel.tsx:314-321`; parent wiring in `TopologyPage.tsx:19-24,32-39,54-116` per the prior audit).

**Correction — `TerminalPanel` already has internal multi-tab management, it is not single-session:**
- `TabDef { id, label, connected }` (`TerminalPanel.tsx:25-29`) and `tabs`/`activeId` state (`TerminalPanel.tsx:322-323`) already exist.
- `addTab()` (`TerminalPanel.tsx:325-331`) and `closeTab()` (`TerminalPanel.tsx:333-341`, with a guard keeping at least one tab open) are already implemented.
- The tab bar UI — status dot, label, close button (shown only when `tabs.length > 1`), "+" button, and a `Ctrl+\`` keyboard shortcut to toggle the whole panel — is already built (`TerminalPanel.tsx:363-435`).
- Each tab renders an independent `TerminalInstance` (`TerminalPanel.tsx:45-310`) that owns **its own** `host`/`username`/`password`/`connState` and **its own** `WebSocket` to `/ws/terminal` (`TerminalPanel.tsx:52-56, 144-190`). All tab instances are kept mounted simultaneously (only `visibility` toggles, `TerminalPanel.tsx:439-457`) so background SSH sessions keep running while a different tab is active — the same "always-mounted" pattern used by `CapturePage`.
- The login form defaults to `SSH_HOST='192.168.18.210'`/`DEFAULT_USER='unmsm'` (`TerminalPanel.tsx:9-10`) but **both fields are editable per tab**, so today's users can already open multiple tabs, each independently authenticating to the same or a different SSH host.

So: there is no single "global SSH terminal" as a distinct entity from a tab-management wrapper — every open terminal, including the first one, already **is** a tab in the same `tabs` array, individually authenticated. The correct framing for Addition 4 is not "add tabs to a single-session component," it is "add a second **connection kind** (Kubernetes `pods/exec` in addition to the existing SSH kind) that can be opened as a new tab inside the pattern that already exists."

### Fact: current state of `CapturePage`'s tab pattern

- `CaptureTab { id, pod, podDisplay, iface }` (`CapturePage.tsx:846-851`), `MAX_CAPTURE_TABS = 8` (`CapturePage.tsx:46`).
- Tab state (`tabs`, `activeTabId`, `splitMode`) is **lifted to the parent** (`Layout.tsx`/`App.tsx`) and passed into `CapturePage` as props with `onTabsChange`/`onActiveTabChange`/`onSplitModeChange` callbacks (`CapturePage.tsx:1224-1237`) — this is a different ownership model than `TerminalPanel`, which owns its `tabs` state internally.
- The tab bar itself (`CapturePage.tsx:1327-1450`+) is hand-rolled inline JSX inside the `CapturePage` function component — it is **not** extracted into a separate reusable component, and it is intertwined with capture-specific concerns: `tabStatuses: Record<string, ConnStatus>` keyed to a packet-capture-specific status enum (`live/paused/error/stopped`, `CapturePage.tsx:1240,1316-1322`), a `nodes` topology query used only to populate the pod/interface add-tab dropdown (`CapturePage.tsx:1255-1259,1288-1289`), and split-view drag-resize state (`splitWidths`, `splitDragRef`, `CapturePage.tsx:1244-1246,1261-1286`) that has no analogue in terminal sessions.

**Conclusion (fact, not opinion): the two tab patterns are architecturally independent implementations of the same idea, and neither currently shares code with the other.** `CapturePage`'s tab bar is the more tightly coupled of the two (state lifted to parent, split-view logic, capture-specific status enum) and is **not a good extraction candidate** for this feature. `TerminalPanel`'s tab bar is self-contained, simpler, and already structurally fits "N independent sessions, each with its own connection lifecycle" — it is the correct base to extend. This directly informs the recommendation in Addition 4.E below: extend `TerminalPanel`'s existing tab model to support a second `kind` of `TerminalInstance`, rather than importing anything from `CapturePage`.

## Addition 1 — Authentication and session management

### A. Current State
**Fact.** There is no authentication anywhere in the stack:
- Frontend: `frontend/src/services/api.ts` — `request()` wrapper hardcodes only `Content-Type: application/json`, never sets `Authorization` (confirmed by prior audit of `api.ts:9-19`). No login page, no protected route, no token storage.
- Backend: `api-server/cmd/server/main.go` registers `corsMiddleware()` (`main.go:62,165-176`) which sets `Access-Control-Allow-Origin: *`, and no auth middleware of any kind is registered on `r` or the `api` route group (`main.go:60-125` per prior audit — no `.Use(authMiddleware())` exists). Every Gin handler in `internal/handlers/*` reads only Gin path/query params, never a request identity.
- WebSocket: the shared `upgrader` used by topology/logs/packets/terminal sets `CheckOrigin: func(r *http.Request) bool { return true }` (`topology.go:19-23`), accepting upgrades from any origin.
- RBAC/cluster identity: the backend's *own* identity toward Kubernetes is a ServiceAccount token (`observer`, `helm/templates/serviceaccount.yaml`) — this authenticates the pod to the cluster, not end users to the application.

### B. Feasibility Verdict
**Moderate risk.**
Technically straightforward (standard JWT/session middleware patterns are well understood), but the risk is **architectural breadth**, not algorithmic difficulty: this system has four distinct entry-point classes (REST, WebSocket, SSH-gateway credentials, and the future k8s-exec gateway in Addition 4) which each need a consistent authorization model, and the frontend currently has zero notion of "current user" to thread through `react-query` cache keys, WebSocket reconnect logic, or the multi-tab capture/terminal state.

### C. What Could Break
- **CORS tightening**: `Access-Control-Allow-Origin: *` (`main.go:167`) must become origin-specific once cookies/tokens are introduced (wildcard + credentials is invalid per the Fetch spec) — any dev-mode Vite proxy setup (`vite.config.ts:14-24`, proxying `/api`→`localhost:8080`) or direct-to-`:8080` connections used by `CapturePage.tsx:840` / `TerminalPanel.tsx:15-17` in dev mode must be re-verified against the new allowed-origin list, or local development breaks silently.
- **WebSocket auth handshake**: all four WS endpoints (`/ws/topology`, `/ws/logs/:ns/:pod`, `/ws/packets`, `/ws/terminal`) currently upgrade unconditionally in the handler before any payload is read. Adding a token check means either (a) validating a token from a query string on the upgrade request (the standard pattern, since custom headers aren't available to browser `WebSocket` clients) or (b) requiring a first-message handshake like `terminal.go` already does for SSH auth (`terminal.go:59-79`). Choosing (a) incorrectly could leak tokens into server access logs and browser history; choosing (b) inconsistently across four handlers risks four subtly different implementations of the same check.
- **The always-mounted multi-tab components** (`CapturePage` per `Layout.tsx:107-122`, `TerminalPanel`'s always-mounted `TerminalInstance`s) each open independent WebSocket connections; if a session expires while 8 capture tabs and several terminal tabs are open, all of them need to fail gracefully and consistently rather than each independently attempting silent reconnect loops (existing reconnect-with-backoff logic lives in `frontend/src/services/websocket.ts`'s `WSManager`, but `CapturePage` and `TerminalPanel` both bypass `WSManager` and use raw `WebSocket` directly — per the prior audit — so a session-expiry policy has to be implemented three times, not once, unless this is unified first).

### D. Latency / Performance Impact
- REST: negligible — one token-validation step per request (HMAC/JWT verify is sub-millisecond) added to Gin middleware, paid once per HTTP call.
- WebSocket: a one-time cost at connection-establishment (validate token before/at upgrade), not a per-message cost — does not touch the packet-forwarding hot path (`api-server/internal/capture/grpc_server.go`'s `publish()`) or the gRPC stream from capture-agent, so **no measurable impact on live packet throughput or capture latency**.

### E. Recommended Implementation Approach
*(Recommendation.)* Add a single `authMiddleware()` Gin middleware validating a bearer token (JWT or opaque session token backed by whatever identity provider the operator already has — out of scope for the app itself to invent) applied to the `api` route group; for WebSocket routes, validate the same token passed as a `?token=` query parameter at upgrade time (matches the pattern `terminal.go` already uses of "read one control message before doing privileged work," just applied at the transport layer instead of as a first JSON frame). Keep this as an **additive middleware layer** — no existing handler function signature needs to change, since Gin middleware injects identity into the request context, and handlers that don't care about identity remain untouched. Do not touch `internal/k8s`, `internal/prometheus`, `internal/loki`, or the gRPC capture pipeline at all — this addition is confined to `cmd/server/main.go` (middleware registration) and a new `internal/auth` package.

### F. Dependencies and Ordering Constraints
**Fact-based reasoning:** authentication tokens transiting an unencrypted channel (current state: Gateway is HTTP-only, `helm/templates/gateway.yaml` per prior audit) are exposed in plaintext to anyone on-path. **Recommendation: Addition 2 (TLS) should land before or simultaneously with Addition 1** — shipping bearer tokens over plain HTTP/WS provides a false sense of security. Addition 4's `pods/exec` feature is explicitly gated on this addition being present first (see Addition 4.F).

## Addition 2 — TLS for internal gRPC and external Gateway

### A. Current State
**Fact.**
- Internal gRPC is plaintext in all four call sites found in the codebase:
  - `capture-agent/internal/grpc/client.go:29-31` — `grpc.NewClient(addr, grpc.WithTransportCredentials(insecure.NewCredentials()), ...)` (capture-agent → api-server, `StreamPackets`).
  - `api-server/internal/capture/grpc_server.go:521` — `grpc.NewClient(addr, grpc.WithTransportCredentials(insecure.NewCredentials()))` (api-server → capture-agent, `EnableTshark`/`DisableTshark`/`Ping`).
  - `capture-agent/internal/control/server.go:25` — `grpc.NewServer()` with no `grpc.Creds(...)` option (capture-agent's control server, plaintext).
  - `api-server/internal/capture/grpc_server.go:621` — `grpc.NewServer()` with no `grpc.Creds(...)` option (api-server's capture-ingest server, plaintext).
- The external Gateway API resource (`helm/templates/gateway.yaml`) defines only an `http`/port-80 listener (per prior full-text audit) — no `https` listener, no `tls` section, no reference to a Kubernetes `Secret` or cert-manager `Certificate`. `README.md:83` documents access as `http://192.168.18.234`, confirming plaintext is the intended current state, not an oversight limited to one file.

### B. Feasibility Verdict
**Low risk** for the Gateway TLS termination (this is infrastructure configuration, not application code). **Moderate risk** for internal gRPC mTLS specifically because of **certificate distribution to a DaemonSet**: capture-agent runs on every node and must each obtain/rotate a valid client certificate, and the connection target for the api-server→capture-agent leg is discovered dynamically at runtime via `<PodIP>:9998` (`grpc_server.go:424-434`, hardcoded port `9998`) — a mTLS setup needs the server certificate's SAN to be valid for a *pod IP*, which is unusual and typically requires either a cert-manager-issued short-lived cert per pod or falling back to a shared cluster-internal CA validating on a non-IP identity (e.g., SPIFFE-style), which is more infrastructure to introduce than the current values.yaml-driven Helm chart has anywhere else.

### C. What Could Break
- If gRPC TLS is misconfigured (e.g., cert/CA mismatch between api-server and capture-agent), the **entire packet-capture pipeline silently stops working** — no packets would reach the ring buffer (`grpc_server.go`'s `publish()`), the frontend would sit in `emptyState: 'waiting'` (`CapturePage.tsx:938-946`) indefinitely with no explicit error surfaced to the user, since `StreamPackets`/`CallEnableTshark` failures are logged server-side but not propagated as a distinct WebSocket error type today.
- If the Gateway TLS listener is added without also updating `HTTPRoute` (`helm/templates/httproute.yaml`) and the frontend's relative-URL assumption (`frontend/src/services/websocket.ts` resolves `ws(s)://<location.host>/ws/...` based on `location.protocol`, per prior audit) breaks nothing by itself — the frontend already correctly derives `wss://` when served over `https://` (`TerminalPanel.tsx:12-13`, `websocket.ts:20-26`) — but any hardcoded `ws://`/`http://` fallback (e.g., dev-mode direct-to-`:8080` connections in `CapturePage.tsx:840`, `TerminalPanel.tsx:15-17`) would need to be confirmed as dev-only and not reachable in the TLS-terminated production path.
- `nginx.conf`'s reverse proxy to `http://observer-api:8080` (`frontend/nginx.conf:14-19,22-29`, per prior audit) is internal cluster traffic (frontend pod → api-server pod) and is unaffected by external Gateway TLS termination unless internal-cluster TLS is also a stated goal — this should be scoped explicitly, since "TLS for the internal gRPC channel" as described only refers to the capture protocol, not the internal HTTP hop from frontend to api-server.

### D. Latency / Performance Impact
- gRPC mTLS: TLS handshake cost is paid once per connection, not per packet batch — `capture-agent`'s `grpc.NewClient` already reconnects lazily and infrequently (`internal/grpc/client.go:51-81`, reopens only on send error), so the handshake overhead is amortized. Per-message overhead after the handshake (AES-GCM encrypt/decrypt) is on the order of low single-digit percent CPU overhead for the packet batch sizes in play (≤50 packets or 100ms flush interval, `capture-agent/internal/capture/manager.go:169,225-227`) — not expected to be user-visible, but should be measured under the capture-agent's existing `GOMEMLIMIT=900MiB`/`GOGC=50` resource envelope (`helm/values.yaml:53-56`, tuned for an already resource-constrained DaemonSet).
- Gateway TLS termination: standard reverse-proxy TLS overhead, paid at the edge, not in the application — negligible relative to the 300ms/2s/5s/10s polling intervals already in the system.

### E. Recommended Implementation Approach
*(Recommendation.)* Split this into two independent, separately-shippable changes:
1. **Gateway TLS** (do first, it's the external attack surface): add a `tls` section to the `Gateway` resource backed by a cert-manager `Certificate`/`Secret` (Cilium's Gateway API implementation supports this natively) — pure Helm-template change, zero Go code touched.
2. **Internal gRPC TLS**: use a **shared cluster-internal CA** (e.g., cert-manager `ClusterIssuer`) issuing a single wildcard-style server certificate valid for the Kubernetes Service DNS name used by api-server (`observer-api`, which is stable, unlike capture-agent's per-pod IP), and for capture-agent, accept connections via **server-side TLS with client cert verification (mTLS) keyed to the ServiceAccount identity**, not the pod IP — avoids the SAN-per-pod-IP problem identified in §B by validating the *client* cert (capture-agent → api-server direction) rather than requiring the api-server to validate a fresh cert per ephemeral agent IP for the reverse (api-server → capture-agent control) direction. For that reverse direction specifically, consider **inverting the control-plane direction** so capture-agent polls/long-polls api-server for `EnableTshark`/`DisableTshark` commands instead of api-server dialing the agent's ephemeral pod IP — this sidesteps the certificate-SAN problem entirely, but is a larger design change than "just add TLS" and should be scoped as a follow-up decision, not bundled into this addition.

### F. Dependencies and Ordering Constraints
No dependency on the other three additions. Can be implemented independently and first, since it touches only transport-layer configuration (`grpc.WithTransportCredentials`, `grpc.Creds`, Helm `Gateway` template) and does not require Addition 1's identity model to exist.

## Addition 3 — Pluggable NF classification / topology edge synthesis

### A. Current State
**Fact.** All classification and edge logic lives in one file, `api-server/internal/k8s/topology.go`, as concrete functions and package-level data, with no interface/strategy abstraction:
- `detectNFType(pod, ifaces)` (`topology.go:219-287`) is a fixed, ordered chain: `app.kubernetes.io/component` label → `nf` label (free5GC convention, handles `iupf`/`psaupf` variants via `formatNFLabel`, `topology.go:136-181`) → `component` label (UERANSIM convention, `formatComponentLabel`, `topology.go:184-193`) → `app.kubernetes.io/name` substring scan against `fallbackNFMap` (`topology.go:196-217,250-259`) → pod-name substring scan with a **free5GC-specific interface heuristic**: `if ifaceNames["n9"] && ifaceNames["n3"] && !ifaceNames["n6"] { return NFTypeIUPF }` (`topology.go:268-271`) — this line hardcodes free5GC's intermediate-UPF (ULCL) interface-naming convention directly into the fallback path.
- `buildEdges(nodes, upfNodeDNNs, dnByDNN)` (`topology.go:923-1044`) hardcodes the free5GC N1/N2/N3/N4/N9/N6/SBI edge-synthesis algorithm as a single function with no seam for a different core's edge rules — notably N3 branches explicitly on `len(iupfs) > 0` (`topology.go:970-978`) and N9 connects *every* iUPF to *every* UPF (`topology.go:992-996`), both encoding free5GC's specific UPF-splitting topology, not the general 3GPP N3/N9 reference points (which don't mandate a UPF split at all).
- `getUPFDNNEntries` (`topology.go:489-520`) parses a ConfigMap key literally named `upfcfg.yaml` with a nested `configuration.dnnList[].dnn` YAML shape (`topology.go:472-482,498`) — this is free5GC's UPF configuration file format verbatim, not a generic schema.
- There is currently **no `NFType` beyond the free5GC/UERANSIM set** (`topology.go:21-40`) and no dispatch mechanism (no registry, no factory, no `TopologyClassifier` interface) — `BuildTopology` (`topology.go:741-789`) calls `podToNode`→`detectNFType` and `buildEdges` directly and unconditionally.

### B. Feasibility Verdict
**Moderate risk.**
Not risky in the sense of breaking anything (this is purely additive if done as an interface extraction), but **moderate effort and moderate design risk**: the danger is under-designing the abstraction boundary. Open5GS's NF set overlaps significantly with free5GC's (AMF/SMF/UPF/NRF/AUSF/UDM/UDR/PCF/NSSF names are shared 3GPP terminology) but its UPF model, ConfigMap/config format, and default Helm chart label conventions differ, so a classifier interface needs to abstract at least three axes: (1) label/name → NFType mapping, (2) edge-synthesis rules (which reference points exist and how they're wired), and (3) DNN/UPF-config parsing — under-scoping to just axis (1) would leave `buildEdges` and `getUPFDNNEntries` still free5GC-only and defeat the purpose.

### C. What Could Break
- If the classifier is made pluggable via a runtime-selected strategy (e.g., `TARGET_CORE=free5gc|open5gs` env var) rather than auto-detected, **every existing deployment must set this correctly or silently regress to no topology at all** — since `resolveNamespaces`/`BuildTopology` currently has a hardcoded fallback to namespace `free5gc` (`topology.go` per prior audit) with no explicit "core type" concept today, introducing one is a new required (or defaulted) piece of configuration that every existing Helm values.yaml must account for.
- `deduplicateDisplayNames` (`topology.go:333-468`) and `buildDNNodes` (`topology.go:525-623`) both operate on the *already-classified* `[]TopologyNode` and are core-agnostic today (they work on `NFType`/`DisplayName`/`Labels` generically) — care must be taken that a refactor doesn't accidentally pull free5GC-specific logic into these currently-generic functions, or pull generic logic out in a way that breaks the free5GC path while adding the Open5GS one.
- The frontend's `topology.ts` types (`NFType` union, `NF_POSITIONS`, `PLANE_COLORS`, `NF_PLANE` static layout constants — per prior audit) hardcode the **same** free5GC NF set on the client side for layout positioning; a backend-only classifier abstraction without a corresponding frontend update would let a new core's NF types flow through as `UNKNOWN` and render incorrectly positioned/uncolored on the topology canvas (`TopologyCanvas.tsx:52-63` fixed X-position table).

### D. Latency / Performance Impact
None expected. Classification and edge synthesis run once per `BuildTopology` call (`topology.go:741`), which is itself only invoked on-demand (`GET /api/topology`) or every 5 seconds by `WatchTopology` (`topology.go:187`) — an interface/strategy dispatch adds a negligible virtual-call indirection compared to the cost already paid by the `Pods(ns).List` Kubernetes API calls in the same function.

### E. Recommended Implementation Approach
*(Recommendation.)* Introduce a small `CoreProfile` interface in a new `internal/k8s/coreprofile` package with three methods mirroring the three axes identified in §B — e.g. `ClassifyNF(pod *corev1.Pod, ifaces []NetworkInterface) (NFType, string)`, `BuildEdges(nodes []TopologyNode, ctx EdgeContext) []TopologyEdge`, and `ParseUPFConfig(cm *corev1.ConfigMap) []DNNEntry` — with the **existing free5GC logic moved verbatim** into a `Free5GCProfile` implementation (a mechanical extraction, not a rewrite, since `detectNFType`/`buildEdges`/`getUPFDNNEntries` are already self-contained functions that just need their signatures adjusted to methods). `BuildTopology` takes a `CoreProfile` as a constructor parameter (dependency injection, matching the pattern already used for `InfraHandler{cs, prom}`/`MetricsHandler{prom, cap, cs}` elsewhere in the codebase, e.g. `infrastructure.go:18-25`, `metrics.go:16-24`) selected once at startup from an env var, defaulting to `Free5GCProfile` to preserve current behavior with zero configuration changes for existing deployments. Explicitly **do not** attempt auto-detection of the core type in this pass — that is a separate, harder problem (heuristically distinguishing Open5GS from free5GC from pod labels alone is itself the kind of fragile string-matching this addition is trying to move away from) and should be a v2 concern if ever needed.

### F. Dependencies and Ordering Constraints
Independent of Additions 1, 2, and 4 — pure backend refactor with no security or transport implications. Can be done in any order relative to the others, though doing it before Addition 4 is mildly convenient only in that both touch `TopologyNode` consumption (Addition 4 reads node/pod identity from the topology view to trigger "Open Shell"), not because of any hard technical dependency.

## Addition 4 — Multi-tab terminal with new Kubernetes-exec session kind

### A. Current State
See the corrected analysis at the top of this document. To restate the key fact: **`TerminalPanel` already supports multiple, independently-managed tabs today** (`TerminalPanel.tsx:25-29,322-341`), each an independent SSH session. The backend `/ws/terminal` handler (`api-server/internal/handlers/terminal.go`) implements exactly one connection kind: on receiving the first WebSocket message, it requires `{"type":"auth","user":...,"password":...}` (`terminal.go:69-79`) and unconditionally proceeds to `ssh.Dial` (`terminal.go:93-108`) — there is no discriminator field for "kind of session," and no code path in this handler touches the Kubernetes API at all. There is **no existing `pods/exec` or `client-go` `remotecommand` usage anywhere in the repository** (confirmed in the prior audit of `internal/k8s` and `internal/handlers` — no `remotecommand.NewSPDYExecutor`, no `PodExecOptions`).

### B. Feasibility Verdict
**Moderate-to-High risk** — split by concern:
- **Frontend tab-model extension**: Low risk. `TerminalPanel`'s existing `TabDef`/`TerminalInstance` split (`TerminalPanel.tsx:25-29,45`) already anticipates one session type per tab instance; adding a `kind: 'ssh' | 'k8s-exec'` discriminator and a second `TerminalInstance`-like component (or a conditional login/connect branch inside the existing one) is additive.
- **Backend exec implementation**: Moderate risk. `client-go`'s `remotecommand` package is a well-established, standard mechanism (used by `kubectl exec` itself), but it has a materially different transport (SPDY/WebSocket-over-HTTP upgrade to the *kubelet*, streamed through the API server) than the raw TCP `ssh.Dial` used today, and needs new error-handling paths (pod not found, container not running, exec permission denied) that don't exist in `terminal.go`'s SSH-only error surface.
- **RBAC**: **High risk if done carelessly** — seeSection on RBAC below. This is the part of Addition 4 that most needs explicit, deliberate scoping rather than a quick addition to the existing ClusterRole.

**Overall verdict: Moderate risk**, contingent on the RBAC scoping recommendation in §E being followed; if `pods/exec` `create` is simply appended to the existing cluster-wide `observer` ClusterRole, the risk becomes **High** for the reasons in §C/§E below.

### C. What Could Break
- **Explicit confirmation requested by the user**: the existing global SSH terminal's behavior, connection flow, and UI placement remain **fully intact and unaffected** by this addition, provided the implementation follows the additive approach in §E — i.e., a new `kind` field defaulting to `'ssh'` for all existing/default-created tabs, a new, separate WebSocket route (not a modification of `/ws/terminal`'s existing wire protocol), and no change to `terminal.go`'s existing SSH-dial code path. If instead the existing `/ws/terminal` route's `termMsg` struct (`terminal.go:19-28`) is modified in place to add exec-specific fields and a shared dispatch branch, that *would* put the working SSH path at risk of regression from an unrelated code change sharing the same function — this is exactly the kind of risk the additive approach avoids.
- If the new k8s-exec tabs reuse `TerminalPanel`'s single shared component (`TerminalInstance`) with an `if (kind==='k8s-exec')` branch deep inside SSH-specific state (`host`/`username`/`password` fields, `connect()` closure at `TerminalPanel.tsx:144-190`), a bug in the new branch's logic (e.g., an unhandled exec error) risks throwing inside a function that also renders the SSH login form, potentially breaking the SSH tab's rendering if they share a single component instance incorrectly — this is avoidable (see §E) but is a concrete regression path if the two kinds are not cleanly separated.
- The Topology view's "Open Shell" trigger needs to pass a `(namespace, pod, container)` triple into `TerminalPanel`, which currently receives only `open`/`onToggle`/`height`/`bodyRef` props (`TerminalPanel.tsx:314-319`) — extending this prop surface must not change the meaning of the existing props or the panel silently stops receiving its open/close/resize signals from `TopologyPage`.

### D. Latency / Performance Impact
- No impact on the packet-capture gRPC pipeline or Prometheus/Loki polling — this feature is fully independent of those data paths.
- The `remotecommand` stream itself has comparable latency characteristics to the existing SSH PTY relay (both are interactive, low-throughput, keystroke-driven WebSocket relays) — no measurable difference expected in steady state. The one-time cost of establishing a `remotecommand` executor (an HTTP upgrade to the API server, which proxies to kubelet, which proxies to the container runtime) is typically higher latency to *first byte* than a direct SSH TCP handshake, but this is a one-time per-session cost, not a recurring one.

### E. Recommended Implementation Approach
*(Recommendation.)*
1. **New, separate backend route**: `GET /ws/exec/:namespace/:pod/:container` implemented in a **new** `internal/handlers/exec.go` file — do not add branches to `terminal.go`. Use `client-go`'s `remotecommand.NewSPDYExecutor` with `PodExecOptions{Stdin, Stdout, Stderr, TTY: true}`, matching the same `{type:"input"|"resize"}`/`{type:"output"}` wire-message shape already used by `/ws/terminal` (`termMsg` in `terminal.go:19-28`) purely for frontend consistency — but as a **separate Go type in the new file**, not a shared/modified struct, so the two handlers can evolve independently without risk of one's change breaking the other.
2. **Frontend**: add a `kind: 'ssh' | 'exec'` field to `TabDef` (`TerminalPanel.tsx:25-29`) and a second, small component (`ExecInstance`, mirroring `TerminalInstance`'s xterm-mounting logic but without the SSH login form — since exec sessions are pre-authorized by the "Open Shell" click itself, not by user-entered credentials) rendered conditionally by `kind` inside the same tab-body mapping (`TerminalPanel.tsx:440-457`) — this reuses the **tab bar and tab-lifecycle code** (`addTab`/`closeTab`/`activeId` at `TerminalPanel.tsx:325-349`) without touching `TerminalInstance`'s SSH-specific internals at all, directly answering the "reuse vs. duplicate" question: reuse the shell (tab strip, panel, lifecycle), do not reuse or branch inside the SSH connection-state component.
3. **Topology trigger**: `TopologyCanvas`'s existing node-click/tooltip action pattern (it already navigates for "Live Capture," `TopologyCanvas.tsx:1057-1059` per prior audit) gets a new "Open Shell" action that calls a new `openExecTab(namespace, pod, container)` callback threaded down from `TopologyPage` into `TerminalPanel` (an added prop, not a change to existing ones) which internally calls the existing `addTab()` with `kind:'exec'` and the target identity pre-filled — no login form is shown for this kind, since Kubernetes RBAC (see below), not user-supplied credentials, is the authorization boundary.
4. **RBAC — explicit recommendation, not a fact**: **do not add `create` on `pods/exec` to the existing cluster-wide `observer` ClusterRole** (`helm/templates/rbac.yaml`). The current ClusterRole is deliberately read-only (`get`/`list`/`watch` only, confirmed in prior audit) and is bound to a ServiceAccount shared by **all three workloads including the static frontend** (`api-server-deployment.yaml:21`, `frontend-deployment.yaml:21`, `capture-agent-daemonset.yaml:21`, per prior audit) — `pods/exec` `create` is categorically more sensitive than any read verb, since it grants arbitrary command execution inside any pod in any namespace the role covers. Appending it to `observer` would mean the **frontend's and capture-agent's ServiceAccount token also carry exec rights**, despite neither needing them — violating least-privilege for no benefit. Recommend instead: a **new, separate ServiceAccount** (e.g., `observer-exec`) and a **namespace-scoped Role** (not ClusterRole) granting `create` on `pods/exec` only in the specific target namespace(s) already configured via `TARGET_NAMESPACES`/`targets[].namespace` (`helm/values.yaml:3-4`) — this bounds the blast radius to exactly the namespaces the tool is meant to observe (typically the 5G-core namespace), not the whole cluster. This new ServiceAccount's token should be mounted **only** into the api-server pod (the one component that will actually call `remotecommand`), never into capture-agent or frontend. This is a small, additive Helm change (`helm/templates/rbac-exec.yaml`, `serviceaccount-exec.yaml`), not a modification of the existing `rbac.yaml`/`serviceaccount.yaml` files, so the existing read-only role's behavior for the other two workloads is provably unchanged.

### F. Dependencies and Ordering Constraints
**Fact-based reasoning, high confidence:** this feature grants **interactive command execution inside cluster pods** — the highest-privilege capability of anything in this system, higher than read-only Prometheus/Loki/K8s queries and arguably higher than the existing SSH terminal (which at least requires the user to already possess valid SSH credentials for the target host; the exec feature as designed requires no credential beyond "can reach the unauthenticated web UI"). **Recommendation: Addition 4 must not ship before Addition 1 (authentication) is in place.** Without Addition 1, *any* anonymous visitor to the web UI could click "Open Shell" on any discovered NF pod and obtain an interactive shell inside a live 5G-core network function — this is a strictly worse exposure than the current SSH terminal, which at least requires the visitor to separately know valid SSH credentials for the configured host. This is not a hard technical dependency (the exec code would function without Addition 1 present) but is a **hard security-sequencing requirement**: implementing Addition 4 without Addition 1 already in production creates an unauthenticated remote-code-execution primitive into the observed cluster.

## G. Consolidated Risk Matrix

| Addition | Risk Level | Estimated Effort | Breaking-Change Potential | Recommended Order |
|---|---|---|---|---|
| 1. Authentication & sessions | Moderate | M | Medium (CORS, WS handshake, dev-mode config) | 2nd |

## H. Recommended Implementation Sequence

1. **TLS first (Addition 2).** It has no dependency on anything else in this list, is the lowest-risk change (especially the Gateway half, which is pure infrastructure config), and establishes the encrypted transport that makes Addition 1's tokens meaningful in transit rather than visible in plaintext.
2. **Authentication second (Addition 1).** Now that transport is encrypted, add identity and session handling; this must be in place before Addition 4 is exposed to any environment beyond a fully trusted network, because Addition 4 introduces exec capability that is unsafe to expose without it.
3. **Pluggable topology classification third (Addition 3).** Fully independent of 1/2/4 from a security-sequencing standpoint; placed third simply because it's pure internal refactor work with no urgency driven by risk, and doing it before Addition 4 means the "Open Shell" trigger in Topology can be wired against a topology model that already anticipates non-free5GC node types, avoiding a second touch to the same trigger code later.
4. **Multi-tab exec terminal last (Addition 4).** Depends on Addition 1 being production-ready (see F above) and benefits from Addition 3 being in place; it is also the addition with the highest inherent privilege, so it should be built and reviewed with the most scrutiny once the safer foundational pieces already exist.

## I. Explicit Non-Recommendations

1. **Do not add `pods/exec` `create` to the existing `observer` ClusterRole.** As detailed in Addition 4.E, this ClusterRole is shared by all three workloads including the frontend, which has no legitimate need for exec rights. The safer alternative — a new, narrowly-scoped, namespace-bound Role and a dedicated ServiceAccount mounted only into api-server — achieves the same functionality with a dramatically smaller blast radius, and is not meaningfully more implementation effort (one extra Helm template file, one extra RBAC binding).
2. **Do not implement Addition 4's "Open Shell" feature before Addition 1 is live in any environment reachable outside a fully trusted, air-gapped network.** If the team's priority is to ship the exec feature *before* authentication work is scheduled, the safer alternative is to gate the feature behind a Helm value that is `false` by default (e.g., `execTerminal.enabled: false`, mirroring the existing `terminal.enabled` pattern at `helm/values.yaml:68`) so it can be merged and tested in trusted lab environments without being live-by-default in any deployment that hasn't yet added authentication.
3. **Do not attempt to build a "generic multi-core" topology abstraction in one pass that tries to also auto-detect the core type.** The two problems (pluggable classification, and *automatic* core detection) are separable, and conflating them risks reintroducing exactly the kind of fragile heuristic string-matching (`strings.Contains(name, "upf")`-style logic, `topology.go:268`) that motivated this addition in the first place. Ship the profile abstraction with an explicit, operator-set configuration value first; revisit auto-detection only if there's a demonstrated need.
